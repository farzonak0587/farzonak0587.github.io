Freedom Project: Wrong Answer Game

Context

For the SEP Freedom Project, we had to figure out how to make something actually work using JavaScript. First, we chose a tool to learn, such as React, jQuery, LESS, or Kaboom, and spent a few months researching and learning how to use it. While learning our tool, we also used JavaScript concepts we learned in class, like arrays, loops, functions, and conditionals.After that, we created an interactive program that used both JavaScript and our chosen tool. We documented our progress on our blogs. The overall goal was to take what we learned and use it to build a functional application from scratch.

What I Made

The main part of my project was a quiz game where the goal is to answer questions incorrectly in order to win. I came up with this idea because I wanted to create a game that helps players think faster and react more quickly under pressure. Since most people are used to trying to get the right answer, forcing them to answer incorrectly makes them stop and think differently. I created an array that stored all of the questions and answers and used JavaScript functions to display them one at a time. I also used conditional statements and if/else statements to check whether the player answered correctly or incorrectly and decide what would happen next. Because every answer was a number, I added input validation to make sure players could only enter numbers. To make the game more challenging, I included a 3-second timer for every question so players had to make decisions quickly instead of overthinking.

After finishing the quiz section, I wanted to add another challenge that focused more on reaction time and quick decision-making, so I created a dodge game using Kaboom.js. In this part of the game, the player controls a blue square using the arrow keys while red blocks fall from the top of the screen. To create the falling obstacles, I used loops to repeatedly generate blocks throughout the game instead of placing them manually. I also used conditional statements to detect collisions between the player and the blocks and determine whether the player won or lost the round. If the player successfully dodged 10 blocks, they moved on to the next quiz question, but if they got hit, the game ended. I spent time adjusting the speed of the falling blocks to make sure the game was challenging without being frustrating, and I added a counter to track how many blocks the player had avoided.

Reflection

Challenges

One of the biggest challenges I ran into while making the dodge game was getting collisions to work properly. At first, my player could move through the red blocks without anything happening, and the “Game Over” text never showed up. I realized that I hadn’t set up a collision area for the player or the blocks, so the game didn’t know when they were touching. To fix it, I learned about the area() function in Kaboom.js, which detects collisions and lets objects interact properly. I added the collision detection to both the player and the blocks, and suddenly the game started recognizing hits. It was actually just a single line of code, but it made all the difference and finally allowed the game to end correctly when the player got hit. I used my learning log and the Kaboom.js website to figure this out, which really helped me understand how collision detection works in games.

Another challenge I faced was trying to prevent players from reusing wrong answers in the quiz. At first, I tried storing all of the answers the player had submitted in an array and checking against it every time they typed a new answer. The problem with this approach was that it made the game way too hard and caused issues when the player first started, because the array assumed there were already answers stored. To fix it, I changed the variable that stored the last answer to null at the beginning of the game. This way, there was no stored answer when the game first started, and it didn’t accidentally block valid input. This solution was much simpler and made the game fairer, while still keeping players from repeating the same answers right away.

Takeaways

  • PatienceBe patient when debugging because even small bugs can take a lot of time to figure out, but sticking with it usually leads to a solution.
  • ResourcesUse notes or other resources to refresh your memory since JavaScript and the tools I used have a lot of concepts that are easy to forget.
  • Flexibility Test different solutions instead of getting stuck on just one idea, because trying different approaches can help you find what actually works faster and more efficiently.

Next Steps

For my next steps, I want to focus on improving the overall gameplay experience and making the project feel more complete. One thing I want to do is add more levels or rounds so the game lasts longer and gets progressively harder, which would make it more challenging and keep players engaged. Another next step is to improve the design and user experience by adding better visuals, animations, or sound effects to make the game feel more polished and less basic.
